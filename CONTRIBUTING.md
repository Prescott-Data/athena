# Contributing to dromos-core

Thank you for contributing to dromos-core. This document covers everything you need to know to open a pull request — from setting up your environment to what reviewers will look for.

---

## Repository Layout

`dromos-core` is a **monorepo** of independent services. Each sub-directory is its own project with its own `.git` history, `Dockerfile`, CI pipeline, and dependencies:

```
dromos-core/
├── athena-memos/          # Go — Memory Operating System (primary service)
├── arango-db/             # Python — Odin Knowledge Graph storage service
├── docintel-chat-agent/   # Python — conversational AI agent
├── docintel-query-agent/  # Python — KG query agent
├── docintel-code-agent/   # Python — code analysis agent
├── docintel-datadiscovery-agent/  # Python — data discovery agent
├── docintel-synthesis-agent/      # Python — synthesis/reporting agent
├── workflow_execution_agent/      # Python — workflow runner
├── rag-pipeline-config/   # Python — experimental RAG pipeline
└── jarvis-score-gamification/     # Gamification scoring
```

> Each service can be developed and deployed independently. Changes to `athena-memos` do not require touching any Python agent, and vice versa.

## Language Conventions

### Go (`athena-memos`)

- **Go version:** 1.24+
- **Formatter:** `gofmt` (run `make lint` before committing)
- **Linter:** `golangci-lint` — config at `.golangci.yml`
- **Module path:** `bitbucket.org/dromos/athena-memos`
- **Generated files:** `api/grpc/gen/*.pb.go` and `*.pb.gw.go` are **not committed**. They are generated by `make generate` (or CI's `Generate Protobuf` step). Do not add them to git.
- **Env vars:** All config lives in `internal/config/config.go`. If you add a new env var, document it there **and** in `README.md §7`.

### Python (`all agents / arango-db`)

- **Python version:** 3.11+
- **Dependency management:** `pip` with `requirements.txt` per service
- **Style:** PEP 8; use type hints on all public functions
- **Async:** `asyncio` + `httpx` for all I/O. Do not use blocking calls in async handlers.

## Branch Strategy

We follow **GitFlow lite**:

| Branch | Purpose | Deploys to |
|---|---|---|
| `main` | Production-ready code | AKS Production |
| `staging` | Pre-release validation | AKS Staging |
| `dev` | Integration of feature branches | Azure Container Apps (dev) |
| `feature/<name>` | Your work | Local / preview |

**Workflow:**
1. Branch off `dev`: `git checkout -b feature/my-change dev`
2. Make your changes, test locally.
3. Open a PR targeting `dev`.
4. After review and merge to `dev`, the CI pipeline deploys automatically.
5. Promotions from `dev` → `staging` → `main` are done by the release manager.

## CI/CD Pipeline

Each service has a `bitbucket-pipelines.yml`. The pipeline for `athena-memos` runs these steps on every push:

1. **Generate Protobuf** — regenerates `api/grpc/gen/*.go` from `api/grpc/memory.proto`
2. **Lint** — `go fmt ./...` + `go vet ./...` + `golangci-lint` + `go mod tidy`
3. **Test** — `go test ./pkg/memoryos ./api/middleware` (unit only; no Docker services needed)
4. **Build** — `CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w"`
5. **Docker build & push** — to the Dromos Azure Container Registry
6. **Deploy** — to the target environment based on branch

Python agent pipelines follow a similar pattern: lint → test → Docker build → deploy.

## Making a Pull Request

### Before You Submit

Run the full local check for whichever service you changed:

**Go (`athena-memos`):**
```bash
make lint        # go fmt + go vet + golangci-lint + go mod tidy
make test-short  # unit tests, no Docker required
make build       # verify the binary compiles
```

**Python agents:**
```bash
pip install -r requirements.txt
# Run any service-specific tests (see service README)
```

### PR Checklist

- [ ] Code compiles / passes lint with zero warnings
- [ ] All existing tests pass (`make test-short` for Go; service test suite for Python)
- [ ] New behaviour is covered by at least one test (unit or integration)
- [ ] New env vars are documented in `README.md` config section and `internal/config/config.go`
- [ ] Protobuf changes: ran `make generate`, did **not** commit the generated `*.pb.go` files
- [ ] If this changes behaviour that could cause data loss or a breaking API change: added a note in `techdebt.md` or discussed in the PR description
- [ ] PR description explains **why** the change is needed, not just what it does

### PR Title Format

```
<service>: <short imperative description>

Examples:
  athena-memos: add exponential backoff to worker retry queue
  docintel-chat-agent: fix nil pointer in P2P receiver shutdown
  arango-db: expose KG extractor as HTTP endpoint
```

## Reporting Technical Debt

If you discover a bug, design flaw, or missing test that you cannot fix immediately, add an entry to `techdebt.md` following the existing pattern:

```markdown
### N. Short Title

**Severity:** High / Medium / Low
**File:** path/to/file.go (function name)

Description of the issue.

**Action:** What should be done to fix it.
```
