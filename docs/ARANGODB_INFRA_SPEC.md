# ArangoDB Setup Guide for Athena (Memory OS)

This document provides the necessary architectural and operational details for provisioning the ArangoDB infrastructure to support Athena's Long-Term Personal Memory (LTPM) knowledge graph. It details the programmatic setup handled by the application, the underlying infrastructure requirements, the execution model for distributed graph analytics, and comprehensive security and monitoring guidelines.

## 1. Application-Driven Initialization (`ltm_setup.go`)

The `memory-os` application embeds autonomous database initialization logic to ensure a consistent and optimized environment. Upon the application's first successful connection to the ArangoDB instance, the `InitializeLTMGraph` routine automatically provisions the entire database structure. 

This programmatic setup guarantees that the database environment is always perfectly synchronized with the application's code-level expectations. It performs the following sequential actions:

1. **Database Creation**: It verifies the existence of the target database (`athena_ltm`). If the database is missing, it dynamically provisions it.
2. **Collection Provisioning**: It scaffolds the full schema required for the memory graph, separating logical entities into distinct document and edge collections.
3. **Index Generation**: It constructs and applies precisely tuned, persistent RocksDB indexes to optimize the specific filtering and sorting patterns generated by our graph analytics engine.

### 1.1 Explicit Collections Managed by the Application
The initialization sequence guarantees the presence of the following collections:

**Document Collections (Graph Nodes):**
- `Identities`: Stores core actor profiles (e.g., specific users, discrete agents like OpenClaw or Athena).
- `Concepts`: Houses abstract knowledge entities, thematic topics, and semantic ideas (e.g., "Agentic AI", "Spaced Repetition").
- `Tools`: Represents external utilities, programming languages, and discrete software packages (e.g., "Golang", "ArangoDB Kubernetes Operator", "Docker").
- `Projects`: Tracks continuous streams of work, user repositories, and distinct initiatives (e.g., "Nexus Protocol", "Colabra API Refactor").
- `Communities`: Dedicated storage for the clustered node groupings generated asynchronously by the Pregel Label Propagation analytics job.

**Edge Collections (Graph Relationships):**
- `MemoryEdges`: Defines the multidimensional relationships connecting the document nodes. Edges contain rich contextual properties, including `heat_score` (relevance over time), `confidence` (certainty of the relation), and specific verbs (e.g., `USES`, `WORKS_ON`, `EXHIBITS`).

### 1.2 Programmatic Index Optimization
To guarantee $O(1)$ and $O(\log N)$ algorithmic time complexity for complex node traversal and analytics calculations, the application strictly enforces persistent indexes. Across every document collection (`Identities`, `Concepts`, `Tools`, `Projects`), the following RocksDB persistent indexes are created and maintained:

1. **`community_id` Target Index**: Enables instant filtering of all nodes belonging to a specific Pregel-calculated community cluster.
2. **`is_bridge` Target Index**: Allows the retrieval engine to immediately isolate critical boundary-spanning concept nodes across the entire graph.
3. **`bridge_score` Target Index**: Supports rapid sorting of bridge nodes by their computed structural importance.
4. **Composite Index `["is_bridge", "bridge_score"]`**: Heavily optimizes the most common analytical query—filtering for bridge nodes and returning them sorted by maximum score—allowing the storage engine to fetch pre-sorted results directly from disk without in-memory sorting overhead.

## 2. Infrastructure Requirements
To provide an uncompromised foundation for Athena's real-time retrieval and background graph analytics workloads, the underlying infrastructure must align with the following specifications:

### 2.1 Storage & Disk I/O Performance
ArangoDB's default RocksDB storage engine is inherently I/O bound during extensive graph traversals. The physical or virtual storage layer must prioritize low latency and high throughput.
- **Volume Type**: Provision Persistent Volumes backed exclusively by SSD or NVMe hardware (e.g., `gp3` on AWS, `Premium SSD` on Azure, or `pd-ssd` on GCP).
- **IOPS (Input/Output Operations Per Second)**: Guarantee a minimum continuous baseline of 3,000 IOPS to support rapid, concurrent multi-hop graph traversals.
- **Storage Engine**: Explicitly verify that `RocksDB` is in use.

### 2.2 Compute Sizing for Distributed Analytics
Athena leverages ArangoDB's distributed **Pregel** processing subsystem to execute heavy graph analytics. Specifically, it uses the Label Propagation algorithm to discover tightly knit semantic communities within the knowledge graph. 

Because Pregel algorithms are synchronous and require loading entire graph topologies into active RAM to perform their supersteps, the compute nodes must be generously provisioned to prevent Out-Of-Memory (OOM) crashes.
- **Memory Allocation**: Provision a baseline request of 16Gi of RAM per DBServer pod, with upper bound limits explicitly set to 32Gi. 
- **CPU Allocation**: Allocate 4 to 8 vCPUs per pod to ensure rapid processing of concurrent AQL (ArangoDB Query Language) traversals.
- **RocksDB Block Cache Tuning**: To ensure the most frequently accessed conversational memory entities are kept hot in RAM, aggressively configure the `--rocksdb.block-cache-size` parameter to span approximately 50% of the pod's total available memory limit.

### 2.3 Kubernetes Execution Strategy
To isolate the performance impact of heavy analytics from real-time agent conversations, the Pregel jobs are explicitly decoupled from continuous execution. 
- **Orchestration**: The graph analytics are triggered via dedicated Kubernetes `CronJob` workloads that interact with Athena's administrative APIs. 
- **Scheduling Windows**: These jobs must be scheduled to run during identified off-peak hours (e.g., 03:00 UTC daily) when active conversational throughput is lowest. 
- **Elastic Scalability**: Utilizing the CronJob pattern allows the Kubernetes scheduler to temporarily spin up high-resource pods specifically to handle the Pregel burst load, and then gracefully scale them back down, ensuring optimal cost-efficiency for the cluster.

## 3. Operations & Security Model

### 3.1 Service Account Authentication
Robust security boundaries are required between the application layer and the database layer. Always utilize dedicated service accounts rather than root-level credentials.
1. **Dedicated User Credential**: Provision a dedicated authentication role named `athena_svc`.
2. **Principle of Least Privilege**: Apply strict access controls. Grant Read/Write (`rw`) database permissions *exclusively* to the isolated `athena_ltm` database. Prevent access to the built-in `_system` database or any adjacent tenants.

### 3.2 Environmental Configuration
The `memory-os` application container expects its connection parameters to be injected securely via the environment. Ensure Kubernetes Secrets mapping to the following variables:
- `ARANGODB_URL`: The fully qualified connection string (e.g., `http://arangodb-cluster:8529`).
- `ARANGODB_USER`: Mapped to the `athena_svc` credential.
- `ARANGODB_PASSWORD`: The secure password for the service account.

### 3.3 Active Monitoring & Alerting
The infrastructure team holds the responsibility for the continuous health of the database layer. ArangoDB exposes robust Prometheus metrics that should be integrated into active Grafana dashboards.

**Alerting Priorities:**
1. **AQL Slow Query Logs**: If any AQL execution time exceeds a 100ms threshold, it is a high-confidence indicator that an application query has degraded into a full-collection scan. This implies a critical index managed by `ltm_setup.go` has been accidentally dropped, corrupted, or bypassed.
2. **Memory Utilization**: Set aggressive alerts if DBServer pod memory consumption exceeds 85% of its hard limit, particularly during the Pregel CronJob execution window.
3. **RocksDB Cache Hit Rate**: A hit rate decaying below 80% indicates the active working set of the graph has outgrown the allocated block cache size and the pod requires vertical scaling.
